## General Questions

### What's new in v4.4?

v4.4 adds four integrated systems:
1. **Memory System** — Persistent context across sessions (90% faster starts)
2. **Sprint Management** — 2-week sprint cycles with velocity tracking
3. **Task Integration** — Native Claude Code task tools (TaskCreate/Update/List/Get)
4. **Bead Method** — Micro-task decomposition (15-45 min increments)

### Do I need to adopt all four systems?

No. You can adopt incrementally:
- **Memory only** — Immediate 90% faster session starts
- **Memory + Sprints** — Add predictable delivery
- **Memory + Sprints + Tasks** — Add structured tracking
- **Full stack** — Add continuous integration via beads

See `UPGRADING_V4.3_TO_V4.4.md` for adoption paths.

### Is v4.4 compatible with v4.3 projects?

Yes. v4.4 is additive:
- All v4.3 features still work
- New directories added (`.claude/memory`, `docs/`)
- New scripts added (`memory-manager.sh`, `sprint-planner.sh`)
- No breaking changes

You can run v4.3 and v4.4 systems side-by-side.

### How long does migration take?

**Memory only:** 10 minutes (see `V4.4_CHEAT_SHEET.md`)
**Full migration:** 4 weeks (see `V4.4_MIGRATION_CHECKLIST.md`)

Each week adds one system. You get benefits immediately, even with partial adoption.

## Memory System

### Where is memory stored?

```
.claude/memory/
├── decisions.json          # Architectural decisions
├── patterns.md            # Learned patterns
├── preferences.md         # Your coding style
├── project-context.json   # Current project state
├── session-history.json   # Session tracking
└── cross-project/         # Shared learnings (optional)
```

### How do I load memory in a session?

**Manual method:**
```
Load memory:
- preferences.md
- project-context.json
- patterns.md
- decisions.json (last 5)
- session-history.json (last session)

Ready to work.
```

**Automated method:**
```bash
./essential/toolkit/memory-manager.sh start
```

### How do I save memory at end of session?

**Manual method:**
```
Update memory:
1. Update project-context.json (current state)
2. Add session to session-history.json
3. Add any new patterns to patterns.md
4. Add any decisions to decisions.json

Save all.
```

**Automated method:**
```bash
./essential/toolkit/memory-manager.sh end
```

### What if I have multiple projects?

Use cross-project memory:

```bash
# Setup global memory
mkdir -p ~/.claude/global-memory

# Sync project memory to global
./essential/toolkit/memory-manager.sh sync

# Import global memory to new project
cd /new/project
./essential/toolkit/memory-manager.sh import
```

Patterns and preferences sync across projects. Decisions and context remain project-specific.

### How does memory reduce session start time?

**Without memory (v4.3):**
- You: Manually explain project state (5 min)
- You: Explain recent work (2 min)
- You: Re-state preferences (1 min)
- You: Explain past decisions (2 min)
- **Total: ~10 minutes**

**With memory (v4.4):**
- Claude: Auto-loads all context (30 sec)
- **Total: 30 seconds**

### Can I edit memory files manually?

Yes! They're just JSON and Markdown:
- `decisions.json` — Valid JSON required
- `patterns.md` — Any markdown
- `preferences.md` — Any markdown
- `project-context.json` — Valid JSON required
- `session-history.json` — Valid JSON required

Edit anytime. Claude reads them at session start.

## Sprint Management

### What's a sprint?

A sprint is a 2-week development cycle:
- **Day 0:** Planning (2 hours)
- **Days 1-10:** Execution (daily 5-min updates)
- **Day 10:** Review (1 hour) + Retrospective (1 hour)

### How do I calculate velocity?

Velocity = average points completed per sprint.

**Example:**
- Sprint 1: 18 points completed
- Sprint 2: 22 points completed
- Sprint 3: 20 points completed
- **Velocity: 20 points/sprint** (average of last 3)

Use this for planning Sprint 4 capacity.

### How do I start my first sprint?

```bash
# Create backlog
./essential/toolkit/sprint-planner.sh backlog

# Edit docs/BACKLOG.md with your stories

# Plan first sprint
./essential/toolkit/sprint-planner.sh plan
```

For your first sprint, start conservative:
- **No velocity history:** Use 15 points
- **Some history:** Use average of last 3 sprints
- **Consistent history:** Use average ± 2 points

### What if I don't finish all stories in a sprint?

**Options:**
1. **Move incomplete stories to next sprint** (recommended)
2. **Extend sprint by 1-2 days** (only if 90%+ complete)
3. **Mark sprint incomplete, retrospective why**

**Do NOT:**
- Artificially mark incomplete work as done
- Skip retrospective
- Plan next sprint without velocity adjustment

### How many points should a story be?

**Guidelines:**
- **1 point:** 1-2 hours (simple bug fix, small feature)
- **2 points:** 2-4 hours (medium feature)
- **3 points:** 4-8 hours (complex feature)
- **5 points:** 8-16 hours (large feature)
- **8 points:** 16-24 hours (very large feature, consider splitting)
- **13+ points:** Too large, must split

If a story is >8 points, decompose into smaller stories.

### Can I change sprint scope mid-sprint?

**Avoid it.** Scope changes cause:
- Inaccurate velocity
- Burndown chart confusion
- Loss of predictability

**If you must:**
1. Document why in sprint notes
2. Mark original scope clearly
3. Track "original points" vs "final points" separately
4. Discuss in retrospective

### What goes in a retrospective?

See `essential/toolkit/templates/RETROSPECTIVE.md` for full template.

**Key sections:**
- What went well (keep doing)
- What went poorly (stop doing)
- Action items (concrete, assignable)
- Velocity analysis (compare to plan)
- Team health (solo devs: your health)

### How do sprints integrate with memory?

**Sprint → Memory:**
- Sprint velocity saved to memory
- Retrospective action items → patterns.md
- Sprint decisions → decisions.json

**Memory → Sprint:**
- Past velocity informs capacity
- Patterns prevent repeated mistakes
- Decisions guide story acceptance

## Task Integration

### What are the native task tools?

Claude Code provides:
- **TaskCreate()** — Create structured tasks
- **TaskUpdate()** — Update status, dependencies
- **TaskList()** — View all tasks
- **TaskGet()** — Get task details

v4.4 integrates these into sprint workflow.

### How do I create a task?

```javascript
TaskCreate({
  subject: "Implement user login endpoint",
  description: "POST /api/login with JWT authentication",
  activeForm: "Implementing user login",
  metadata: {
    sprint: "7",
    story: "User Authentication",
    points: 3
  }
})
```

### How do task dependencies work?

```javascript
// Task 2 depends on Task 1
TaskUpdate({
  taskId: "2",
  addBlockedBy: ["1"]
})

// Task 1 blocks Tasks 2 and 3
TaskUpdate({
  taskId: "1",
  addBlocks: ["2", "3"]
})
```

When Task 1 completes, Tasks 2 and 3 automatically unblock.

### What's the task lifecycle?

```
pending → in_progress → completed
         ↓
       deleted (if cancelled)
```

**Always:**
- Set `status: "in_progress"` when starting
- Set `status: "completed"` when done
- Use TaskList() to find next available task

### How do tasks integrate with sprints?

**Sprint planning:**
- Break stories into tasks (TaskCreate)
- Set dependencies (TaskUpdate with blockedBy)

**Sprint execution:**
- Update status daily (TaskUpdate)
- Track completion via TaskList()

**Sprint review:**
- Verify all tasks completed
- Incomplete tasks move to next sprint

### Can I use tasks without sprints?

Yes. Tasks work standalone:
- Create ad-hoc tasks with TaskCreate()
- Track progress with TaskList()
- Update as you work

Sprints add structure, but tasks are independent.

## Bead Method

### What's a bead?

A bead is a micro-task with these properties:
- **Time:** 15-45 minutes
- **Output:** Working code
- **Commit:** One commit per bead
- **Value:** Incrementally valuable

### Why use beads?

**Benefits:**
- Always working code (can stop anytime)
- Continuous integration (no big bang merges)
- Visible progress (8/12 beads done)
- Low sunk cost (easy to pivot)
- Reduced stress (clear, achievable goals)

### How do I decompose a feature into beads?

**Example: User Authentication (8 hours)**

**Traditional approach:**
- Write all code (8 hours)
- Commit once
- Risk: Non-working code if interrupted

**Bead approach:**
1. Create User model (15 min) → commit
2. Add password hashing (20 min) → commit
3. POST /register stub (20 min) → commit
4. Email validation (15 min) → commit
5. Password validation (15 min) → commit
6. POST /login stub (20 min) → commit
7. JWT generation (30 min) → commit
8. Token validation (25 min) → commit
9. Auth middleware (30 min) → commit
10. Integration test (30 min) → commit
11. Error handling (20 min) → commit

**Result:** 11 beads, ~4 hours, 11 commits, always working

### What if a bead takes longer than 45 minutes?

**Stop and reassess:**
1. Is the bead too large? Split it
2. Did you hit a blocker? Document and switch beads
3. Is the estimate wrong? Update future estimates

**Do NOT:**
- Keep working past 1 hour without committing
- Force completion of oversized bead
- Skip commit because "it's not done"

Commit working state, create new bead for remaining work.

### How do beads integrate with tasks?

**Task = Collection of Beads**

Example:
```
Task: "Implement user registration"
  ├── Bead 1: User model
  ├── Bead 2: Password hashing
  ├── Bead 3: POST /register stub
  ├── Bead 4: Email validation
  └── Bead 5: Password validation

Task completes when all beads done.
```

### What if I'm interrupted mid-bead?

**Commit what you have:**
```bash
git add .
git commit -m "WIP: Partial implementation of user login

Completed:
- Basic endpoint structure
- Request validation

Still needed:
- JWT generation
- Error handling

Co-Authored-By: Claude Sonnet 4.5 <noreply@anthropic.com>"
```

Add remaining work as new bead. Resume when ready.

### How many commits per day should I expect?

**With beads:** 8-12 commits/day

**Calculation:**
- 6 productive hours/day
- Bead duration: 15-45 min average (30 min)
- Expected: 6 hours / 0.5 hours = 12 commits
- Realistic (meetings, breaks): 8-10 commits

**If you're doing fewer commits:**
- Beads too large (split them)
- Interruptions too frequent (time-box work)
- Overthinking (commit working code)

## Integration & Workflow

### How do all four systems work together?

**Week 1 (Planning):**
1. Sprint planning → Create stories
2. Break stories → Create tasks (TaskCreate)
3. Decompose tasks → Create bead chains
4. Commit sprint plan

**Week 2 (Execution):**
```bash
# Daily workflow
./essential/toolkit/memory-manager.sh start    # Load context
# ... execute beads (commit after each)
# ... update tasks (TaskUpdate)
./essential/toolkit/sprint-planner.sh update   # Track progress
./essential/toolkit/memory-manager.sh end      # Save context
```

**Week 2 End (Review):**
1. Sprint review (demo, accept stories)
2. Retrospective (learnings)
3. Learnings → Memory system
4. Velocity → Next sprint

### What's a typical daily workflow?

**Morning (5 min):**
```bash
./essential/toolkit/memory-manager.sh start
```
Claude loads context, you're ready to work.

**During day:**
- Execute beads (15-45 min each)
- Commit after each bead
- Update task status (TaskUpdate)
- Take breaks between beads

**Afternoon:**
```bash
./essential/toolkit/sprint-planner.sh update
```
Update sprint progress, check burndown.

**Evening (5 min):**
```bash
./essential/toolkit/memory-manager.sh end
```
Save context for tomorrow.

### Do I need to use all the scripts?

**Minimum viable workflow:**
- Memory: Manual (paste load/save prompts)
- Sprints: Manual (edit SPRINT.md files)
- Tasks: Use native tools (no scripts)
- Beads: Commit discipline (no scripts)

**Recommended workflow:**
- Memory: `memory-manager.sh` (saves 5 min/session)
- Sprints: `sprint-planner.sh` (saves 10 min/update)
- Tasks: Native tools
- Beads: Discipline + bead chain templates

**Power user workflow:**
- Automate everything with scripts
- Add git hooks for auto-memory save
- Create session starter alias
- Dashboard for sprint metrics

## Troubleshooting

### "Memory files not loading"

**Check:**
```bash
ls -la .claude/memory/
```

**Fix:**
```bash
# Create if missing
mkdir -p .claude/memory

# Copy templates
cp $CLAUDE_HOME/claude-code-framework/essential/toolkit/templates/memory/* .claude/memory/

# Or create manually (see V4.4_CHEAT_SHEET.md)
```

### "Sprint velocity is unstable"

**Causes:**
- Inconsistent estimation
- Scope creep mid-sprint
- Interruptions/distractions
- Team size changes

**Fix:**
- Use consistent estimation (planning poker)
- Lock sprint scope on Day 0
- Time-box work sessions
- Track actual vs estimate per story

### "Tasks show wrong dependencies"

**Check:**
```javascript
TaskGet({ taskId: "X" })
// Shows blockedBy and blocks arrays
```

**Fix:**
```javascript
// Clear all dependencies
TaskUpdate({ taskId: "X", addBlockedBy: [] })

// Re-add correct dependencies
TaskUpdate({ taskId: "X", addBlockedBy: ["Y"] })
```

### "Beads taking too long"

**Common causes:**
- Bead too large (should be 15-45 min)
- Blocker encountered (document, switch beads)
- Context switching (time-box sessions)

**Fix:**
- Split large beads into smaller ones
- Use bead chain template for planning
- Measure actual time, adjust estimates

### "Memory growing too large"

**Limits:**
- `decisions.json`: Keep last 20-30 decisions
- `session-history.json`: Keep last 10-20 sessions
- `patterns.md`: Archive old/irrelevant patterns

**Cleanup:**
```bash
./essential/toolkit/memory-manager.sh clean
```

Moves old entries to `.claude/memory/archive/`.

## Best Practices

### Memory System

✅ **Do:**
- Update memory at session end
- Add decisions as you make them
- Document patterns after 2nd occurrence
- Keep preferences current

❌ **Don't:**
- Let memory files grow unbounded
- Store passwords/secrets in memory
- Skip session-history updates
- Forget to sync cross-project learnings

### Sprint Management

✅ **Do:**
- Lock sprint scope on Day 0
- Update progress daily
- Do retrospectives (even solo)
- Track velocity over 3+ sprints

❌ **Don't:**
- Skip retrospectives
- Change scope mid-sprint
- Ignore velocity trends
- Plan beyond capacity

### Task Integration

✅ **Do:**
- Set dependencies upfront
- Update status as you work
- Use metadata for sprint/story tracking
- Mark completed when truly done

❌ **Don't:**
- Leave tasks in "in_progress" forever
- Skip TaskList() checks
- Ignore blockedBy warnings
- Create tasks without descriptions

### Bead Method

✅ **Do:**
- Commit after each bead
- Keep beads 15-45 min
- Maintain working code
- Update bead chain progress

❌ **Don't:**
- Work >1 hour without committing
- Commit broken code
- Skip bead planning
- Ignore time estimates

## Migration & Adoption

### Should I migrate existing projects to v4.4?

**Migrate if:**
- Active development (2+ sessions/week)
- Multi-week timeline
- Complex project (>5k LOC)
- Team collaboration

**Skip if:**
- One-off scripts
- Nearly complete projects
- Simple prototypes
- Archived projects

### Can I adopt v4.4 mid-project?

Yes. Start with memory system:

**Week 1:**
1. Create `.claude/memory/` directory
2. Initialize memory files with current state
3. Use memory for all sessions

**Week 2:**
4. Create backlog from remaining work
5. Plan Sprint 1 with conservative estimate
6. Track velocity

**Week 3+:**
7. Add task integration
8. Adopt bead method

### How do I train a team on v4.4?

**Day 1 (2 hours):**
- Read: `V4.4_RELEASE_SUMMARY.md`

- Setup: Memory system for each dev

**Week 1:**
- Practice: Memory system usage
- Measure: Session start time improvement

**Week 2:**
- Learn: Sprint planning together
- Execute: First sprint (pair if needed)

**Week 3+:**
- Adopt: Tasks and beads
- Review: Sprint retrospectives together

## Costs & Performance

### Does v4.4 increase API costs?

**Memory system:** No. Reading memory files is free.
**Sprint management:** No. Local markdown files.
**Task integration:** Minimal. Task tools are lightweight.
**Bead method:** No. Just commit discipline.

**Net impact:** Neutral to slightly negative (more efficient sessions = less wasted tokens)

### How much time does v4.4 save?

**Per session:**
- Memory system: 9-14 minutes saved at start

**Per sprint:**
- Sprint management: 2-4 hours saved (vs unstructured work)
- Task integration: 30-60 min saved (vs manual tracking)
- Bead method: 1-3 hours saved (vs debugging big bang commits)

**Per month (~4 sprints):**
- ~40 sessions × 10 min = 400 min (6.6 hours)
- ~4 sprints × 3 hours = 12 hours
- **Total: ~18 hours saved/month**

### What are the storage requirements?

**Memory system:** <1 MB per project
**Sprint docs:** ~100 KB per sprint
**Task data:** Claude Code manages (minimal)
**Bead chains:** ~50 KB per feature

**Total:** <10 MB per project over 6 months

Negligible storage impact.
